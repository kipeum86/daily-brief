## Problem Statement

한국 투자자/직장인이 매일 아침에 알아야 할 것들이 너무 흩어져 있다. 시장 현황은 네이버금융, 세계 뉴스는 뉴스 앱, 국내 정치는 또 다른 곳. 한 곳에서 "오늘 내가 알아야 할 모든 것"을 30초 만에 파악할 수 있는 도구가 없다.

Macro-Pulse(매크로 지표 봇)의 아이디어에서 출발하되, The Economist "World in Brief" 스타일의 글로벌/국내 뉴스 브리핑까지 결합한 **개인 큐레이티드 데일리 브리프**.

## What Makes This Cool

매일 아침 하나의 브리핑에 모든 것이 담긴다:
- **Morning Brief** (KST 07:30): 한국 장 전일 마감 + 미국 장 마감 + 세계 뉴스 + 한국 뉴스 — "출근 전 3분에 모든 것"

단순 수치 나열이 아니라 **AI가 맥락을 읽어준다**:
- "VIX 급등 + 달러 강세 → 위험자산 회피 신호"
- "어제 미 연준 발언과 오늘 원/달러 움직임의 연결고리"
- "중동 정세 긴장 → 유가 영향 → 한국 시장 임팩트"

TradingView보다 예쁘고, 네이버금융보다 똑똑하고, Economist보다 한국 투자자에게 맞춤.

## Constraints

1. **1인 개발** (Claude Code + AI 보조)
2. **서버 비용 $0** — GitHub Actions(스케줄 실행) + GitHub Pages(대시보드 호스팅)
3. **데이터 소스 무료** — Yahoo Finance, FRED, RSS 피드
4. **AI 비용 최소화** — Claude API 호출, 하루 1회 (아침)
5. **기존 코드 재활용** — auto-newsbriefing의 RSS/분류 패턴, stock-analysis-agent의 데이터 수집 참고
6. **새 프로젝트** — auto-newsbriefing과 별개 레포로 생성

## Premises

1. 하루 1회 아침 브리핑이 실시간보다 가치 있다 — 정보 과잉 시대에 큐레이션이 핵심
2. 무료 API(Yahoo Finance, FRED, RSS)로 충분한 품질의 데이터를 수집할 수 있다
3. AI 해설이 핵심 차별점이다 — 수치만 보여주는 건 네이버금융과 동일
4. GitHub Actions + Pages로 서버 없이 운영 가능하다
5. 웹 대시보드 + 이메일이 최적의 채널 조합이다 (텔레그램/카카오톡 불필요)

## Approaches Considered

### Approach A: Static Dashboard Generator (Minimal Viable)
GitHub Actions가 하루 2회 데이터 수집 → AI 해설 생성 → 정적 HTML 대시보드를 GitHub Pages에 배포 + 이메일 발송. Chart.js로 인터랙티브 차트.

- **Effort:** M (human: ~2주 / CC: ~3-4시간)
- **Risk:** Low
- **Pros:**
  - 서버 비용 $0, GitHub만으로 전부 가능
  - 가장 빠르게 실사용 가능
  - 정적이지만 Chart.js로 인터랙티브 차트 가능
- **Cons:**
  - 실시간 업데이트 불가 (하루 2회 갱신)
  - 프론트엔드 커스터마이징에 한계
- **Completeness:** 7/10

### Approach B: Python + Jinja2 + Chart.js + GitHub Actions/Pages (Ideal)
Python으로 데이터 수집 + Claude API 해설 → Jinja2 템플릿으로 아름다운 HTML 대시보드 생성 (섹션별: Markets, World, Korea, AI Insight) → GitHub Pages 배포. 이메일도 같은 Jinja2 템플릿에서 HTML 이메일 생성. Chart.js/Plotly.js로 히트맵 + 인터랙티브 차트.

- **Effort:** M-L (human: ~3주 / CC: ~4-5시간)
- **Risk:** Low
- **Pros:**
  - Jinja2 템플릿으로 웹+이메일 동시 생성 가능
  - Plotly.js 히트맵으로 포트폴리오급 비주얼
  - 섹션별 모듈화로 확장 용이
  - 기존 Python 프로젝트들과 일관성
- **Cons:**
  - A보다 초기 세팅 약간 더 필요
  - 히트맵 구현: Plotly.js treemap 사용 (D3.js보다 쉽고, Finviz 스타일 히트맵 바로 가능)
- **Completeness:** 9/10

### Approach C: Next.js + Vercel (Over-engineered)
Next.js SSR 대시보드, Vercel 배포, 실시간 데이터 fetch.

- **Effort:** L-XL (human: ~5주 / CC: ~6-8시간)
- **Risk:** Med-High
- **Pros:** 실시간, 가장 기술적으로 인상적
- **Cons:** 과도한 복잡도, Vercel 비용, 하루 2회 브리핑 컨셉과 맞지 않음
- **Completeness:** 9/10 (but over-engineered for the use case)

## Recommended Approach

**Approach B: Python + Jinja2 + Chart.js/Plotly.js + GitHub Actions/Pages**

이유:
- 하루 2회 정적 생성이 컨셉에 완벽히 맞음 (실시간 불필요)
- Jinja2로 웹 대시보드와 이메일 HTML을 하나의 템플릿 시스템에서 생성
- Python이라 기존 프로젝트(auto-newsbriefing, stock-analysis-agent)의 패턴 재활용
- Plotly.js 히트맵 + Chart.js 차트로 포트폴리오급 비주얼 달성
- 서버 비용 $0 유지

## 코드 재활용 전략: Copy-and-Own

auto-newsbriefing에서 다음 모듈을 복사하여 daily-brief에 맞게 독립적으로 유지:
- **복사**: collector.py (RSS), dedup.py, llm/base.py + claude.py, config.py, models.py, keyword_filter()
- **전략**: copy-and-own — 복사 후 daily-brief 요구사항에 맞게 자유롭게 수정. auto-newsbriefing과 동기화하지 않음.
- **근거**: solo 프로젝트에서 공유 패키지 추출은 과도한 엔지니어링. 두 프로젝트의 요구사항이 다르므로 독립 진화가 자연스러움.

## Eng Review 반영 사항 (2026-03-24)

- **yfinance fallback**: yfinance 실패 시 자동 fallback (Yahoo Finance v8 REST 또는 Alpha Vantage)
- **통합 렌더 파이프라인**: 하나의 데이터 준비 → dashboard HTML + email HTML 동시 생성 (DRY)
- **Chart.js only v1**: Plotly.js treemap 히트맵은 v2로 연기. v1은 Chart.js 바/라인 차트
- **1회 통합 Claude API 호출**: 시장 데이터 + 뉴스 제목을 한 번에 보내서 통합 브리핑 생성 (하루 1회, 월 ~$0.3)
- **Config 검증**: main.py 시작 시 config.yaml 필수 필드 검증 + 명확한 에러 메시지
- **병렬 수집**: ThreadPoolExecutor로 RSS + 시장 데이터 병렬 fetch (~30초→~5초)

## Resilience & Error Handling

- **데이터 소스 실패 시**: 해당 섹션만 "데이터 수집 실패" 표시, 나머지 섹션은 정상 생성 (graceful degradation)
- **yfinance 불안정성**: yfinance는 비공식 스크래퍼로 Yahoo가 변경 시 깨질 수 있음. 대안: Alpha Vantage 무료 tier (US 데이터), KRX Open API (한국 데이터)
- **RSS 피드 실패 시**: 최근 캐시된 뉴스로 대체 + "업데이트 실패" 표시
- **Claude API 실패 시**: AI 인사이트 섹션 생략, 데이터만으로 브리핑 생성
- **재시도**: 각 소스 최대 3회 재시도 (exponential backoff)
- **한국 RSS 피드**: 연합뉴스, 한경 등은 피드 안정성이 낮을 수 있음 → 네이버 뉴스 RSS를 안정적 대안으로 우선 사용
- **FRED API**: 하루 1000건 제한이지만, 브리핑당 ~5개 시리즈 조회 = 하루 10건 이하 — 제한에 여유 있음
- **Jinja2 렌더링 실패 시**: 렌더링 결과 최소 길이 검증 — 빈/깨진 HTML 배포 방지

## 주말/공휴일 캘린더 규칙

- **cron**: 평일(월-금)만 실행 — 기존 설계 유지
- **시장 데이터**: 장이 열린 날만 해당 시장 섹션 표시. 휴장일에는 "오늘 [KOSPI/NYSE] 휴장" 안내 배너 표시
- **뉴스**: 매일 수집 — 시장 휴장 여부와 무관하게 뉴스+AI 인사이트는 항상 생성
- **한국 공휴일**: KOSPI 휴장 → 한국 시장 섹션에 "오늘 KOSPI 휴장 (삼일절)" 등 안내. 미국 시장/뉴스는 정상
- **미국 공휴일**: NYSE 휴장 → 미국 시장 섹션에 "오늘 NYSE 휴장 (Memorial Day)" 등 안내. 한국 시장/뉴스는 정상
- **월요일 아침 브리핑**: 금요일 미국 장 마감 데이터 + 주말 뉴스 커버. 뉴스 top_n을 평소보다 늘려서 (예: 5→8) 주말 뉴스 충분히 반영
- **휴장 판별**: yfinance에서 데이터가 전일과 동일하면 휴장으로 판단 (별도 캘린더 API 불필요)

## Cross-run 중복 제거

- auto-newsbriefing의 `trends/` 패턴 활용: 각 브리핑 실행 후 처리된 기사 URL/EventKey를 `trends/trend_YYYY-MM-DD.txt`에 저장
- 다음 날 브리핑 실행 시 최근 트렌드 파일 로드 → 어제와 동일 기사 중복 방지
- Weekly/Monthly recap 생성 시에도 트렌드 파일 참조하여 이미 다룬 뉴스 재활용 방지
- GitHub Actions에서 trends/ 파일 유지: GitHub Actions cache 사용

## Secrets Management

GitHub Actions Repository Secrets에 저장:
- `ANTHROPIC_API_KEY`: Claude API
- `FRED_API_KEY`: FRED 경제 데이터
- `EMAIL_API_KEY`: Resend API 키 (또는 Gmail SMTP 비밀번호)
- `GOOGLE_SHEETS_CREDENTIALS`: Google Sheets 서비스 계정 JSON (base64 인코딩)
- (수신자 목록은 config.yaml에서 관리 — 별도 시크릿 불필요)

## Email Subscriber Management

- 수신자 목록은 `config.yaml`의 `subscribers` 섹션에 관리 (소규모 5-10명)
- 추가/제거는 config.yaml 수정 후 commit
- 대규모 확장 시 Resend의 Audience 기능 또는 별도 서비스 도입 검토

## Data Sources (무료)

### 시장 데이터
- **Yahoo Finance** (yfinance): 주가, 환율, 원자재, 암호화폐
- **FRED API**: 미 국채 수익률, 달러 인덱스, 매크로 지표
- **한국은행 ECOS API**: 한국 경제 지표 (10x Vision — Phase 2에서 검토)

### 뉴스 RSS
- **글로벌**: Reuters, AP News, BBC World, The Guardian
- **한국**: 연합뉴스, 한국경제, 조선일보, 매일경제
- **경제/금융**: CNBC, MarketWatch (Bloomberg/FT 무료 RSS는 헤드라인만 제공, 본문 없음)

### AI (Pluggable LLM Provider)
- **config.yaml에서 provider 선택**: `gemini` (기본), `claude`, `openai`
- auto-newsbriefing의 pluggable provider 패턴 복사 (base.py + 각 provider 구현)
- 기본 provider: **Gemini** (무료 tier 넉넉, 하루 1회 호출에 충분)
- 호출 구조: 1회 통합 호출 (시장 데이터 + 뉴스 제목 → 통합 브리핑 + 인사이트) = 하루 1회
- 예상 토큰: 입력 ~3,000 tokens, 출력 ~800 tokens
- 예상 비용: Gemini 무료 tier 내 (월 $0)

### 정보 위계 원칙
- **AI Insight 맨 위**: 이 브리핑의 유일한 차별점. 숫자는 네이버금융에도 있지만, AI 교차분석은 여기에만 있음
- **선형 스캔**: 모든 섹션이 같은 폭, 위→아래 순서. 그리드/탭/사이드바 없음
- **숫자는 컴팩트**: Markets 섹션은 최소 공간에 최대 정보 밀도
- **뉴스는 스캔 가능**: 제목 + 1줄 요약, 원문 링크

### 과거 브리핑 탐색
- **웹**: 헤더에 ◀▶ 날짜 네비게이션 → archive/{date}.html로 이동
- **/archive 페이지**: 날짜 목록 + Weekly/Monthly recap 링크

### 디자인 원칙
1. **컬러는 데이터에만** — 상승/하락만 컬러, 나머지 모노톤
2. **AI 해설이 시각적 주인공** — serif 폰트 + warm beige 배경으로 다른 섹션과 구분
3. **숫자는 mono** — 시장 수치는 고정폭 폰트로 정렬감 확보

5. **구분은 선으로** — 카드가 아닌 1px 구분선으로 섹션 분리 (Economist 스타일)

## User Journey

1. **아침 이메일 수신** → 제목: "Daily Brief · 3/24 — VIX 급등, 위험자산 회피 신호"
2. **이메일 상단 CTA**: "🌐 웹에서 보기" 버튼 → 당일 웹 대시보드 링크
3. **이메일에서 AI Insight 읽기** → 에디토리얼 문단으로 오늘의 맥락 파악
4. **숫자 스캔** → matplotlib PNG 차트 (이메일) / Chart.js 인터랙티브 (웹)
5. **뉴스 스캔** → 제목 + 1줄 요약, 원문 링크
6. **웹 전환** → CTA 클릭 시 GitHub Pages 대시보드, Chart.js 인터랙티브 차트
7. **과거 탐색** → ◀▶ 날짜 네비 + /archive 페이지
8. **공유** → 웹 푸터에 URL 복사 + "친구에게 추천하기" 링크 (구독 유도)

## Weekly & Monthly Recap (v1.5)

### Weekly Recap (매주 금요일 저녁 또는 토요일 아침)
- **cron**: `'0 0 * * 5' (UTC)` = KST 토요일 09:00
- **내용**:
  - 주간 시장 등락 요약 (한국 + 미국 주요 지수 주간 변동률)
  - 주간 Top 뉴스 3-5건 (가장 임팩트 있었던 이벤트)
  - AI 주간 인사이트: "이번 주 핵심 테마" + "다음 주 주목 포인트"
  - Google Sheets에서 주간 시계열 데이터 가져와 Chart.js 주간 차트 생성
- **데이터 소스**: Google Sheets 아카이브 (일일 데이터 집계) + RSS (주간 종합)

### Monthly Recap (매월 1일 아침)
- **cron**: `'30 22 1 * *' (UTC)` = KST 매월 2일 07:30
- **내용**:
  - 월간 시장 성과 (한국/미국 지수, 환율, 원자재 월간 변동률)
  - 월간 주요 이벤트 타임라인
  - AI 월간 인사이트: "이번 달 키워드" + "다음 달 전망"
  - 월간 Chart.js 차트 (일별 지수 추이)
- **데이터 소스**: Google Sheets 아카이브 (일일 데이터 집계)

### 구현 순서
1. v1: 일일 브리핑 + Google Sheets 저장 (핵심 파이프라인 안정화)
2. v1.5: Weekly/Monthly recap 추가 (Sheets 데이터 기반, 별도 GitHub Actions workflow)

## What I noticed about how you think

- "전부 다"를 여러 번 선택했어 — 비전이 크고 타협하고 싶지 않아 한다. 좋은 본능이야. AI+gstack 시대에 "전부 다"가 실제로 가능하니까.
- Macro-Pulse를 보고 "내 취향에 맞게 새로 만들 수 있지 않을까?"라고 했어 — 기존 것을 그대로 쓰지 않고 자기만의 버전을 만들려는 builder 본능.
- 스콥을 자발적으로 넓혔어 (경제 → 정치/사회/국제 뉴스). "Economist World in Brief 느낌으로"라는 레퍼런스 — 좋은 제품 감각. 최고 수준의 레퍼런스를 기준으로 삼는다.
- 카카오톡 채널에 도전하겠다고 했다가, 핵심이 뭔지 다시 물어보며 방향을 잡았어 — 기능에 휘둘리지 않고 프로덕트 정체성을 먼저 확립하려는 사고방식.
